## TL;DR

项目启动 = 明确目标 + 定义范围 + 组建团队 + 制定计划 + 召开启动会

## 1. 项目启动认知

### 1.1 为什么启动重要
```
项目启动的重要性：
├── 奠定基础：好的开始是成功的一半
├── 统一认知：让所有人对项目有一致理解
├── 建立信任：让团队建立协作关系
├── 预防问题：提前识别潜在风险
├── 明确责任：让每个人知道自己的职责
└── 获得支持：让干系人认可和支持项目
```

### 1.2 启动常见问题
```
项目启动常见问题：
├── 目标不清晰：不知道要达成什么
├── 范围不明确：做到什么程度算完成
├── 责任不清：不知道谁负责什么
├── 计划不实：计划太理想化
├── 资源不足：人、钱、时间不够
├── 干系人没对齐：各方期望不一致
└── 启动会流于形式：开完会没收获
```

### 1.3 启动成功标准
```
启动成功的标志：
├── 目标清晰：所有人都知道要达成什么
├── 范围明确：知道项目做什么、不做什么
├── 团队就绪：团队成员知道自己角色
├── 计划可行：有切实可行的项目计划
├── 资源到位：必要的资源已到位
├── 干系人支持：关键干系人认可项目
└── 有明确的沟通机制：知道怎么沟通汇报
```

## 3. 干系人分析

### 3.1 干系人识别
```
干系人识别方法：

内部干系人：
├── 项目发起人：项目发起/支持的人
├── 项目经理：项目负责人
├── 核心团队：执行项目的人
├── 职能部门：相关部门负责人
├── 高层管理：关注项目进展的人
└── 其他项目团队：可能有协作的团队

外部干系人：
├── 客户/用户：最终使用者
├── 合作伙伴：合作方
├── 供应商：提供资源/服务
├── 监管机构：监管/审批部门
└── 其他外部：可能受影响的外部方
```

### 3.2 干系人分类
```
干系人分类矩阵：

        高影响力    低影响力
高关注  ┌─────────┬─────────┐
        │ 重点管理  │  保持知悉  │
        │（发起人） │（部分成员）│
利益    ├─────────┼─────────┤
低关注  │  随时告知  │  监督    │
        │（监管机构）│（一般成员）│
        └─────────┴─────────┘

管理策略：
├── 重点管理：高权力+高关注，全力满足
├── 保持知悉：高权力+低关注，定期汇报
├── 随时告知：低权力+高关注，及时通知
└── 监督：低权力+低关注，适度关注
```

### 3.3 干系人沟通计划
```
干系人沟通计划：

| 干系人 | 关注点 | 沟通方式 | 频率 | 负责人 |
|--------|--------|----------|------|--------|
| 发起人 | 进度/风险 | 周报+当面 | 每周 | 项目经理 |
| 高层管理 | 里程碑/决策 | 汇报+邮件 | 双周 | 项目经理 |
| 核心团队 | 任务/协作 | 站会+文档 | 每日 | 项目经理 |
| 客户 | 需求/交付 | 演示+沟通 | 按需 | 产品经理 |
```

## 4. 项目范围定义

### 4.1 范围定义方法
```
范围定义方法：

用户故事法：
├── 作为XX用户
├── 我想要XX功能
└── 以便于XX价值

功能列表法：
├── 核心功能（必须做）
├── 重要功能（应该做）
├── 次要功能（可以做）
└── 明确不做（明确排除）

边界分析法：
├── 明确边界：什么是项目范围内的
├── 识别模糊地带：可能有争议的地方
├── 定义交接点：与其他系统/团队的边界
└── 文档化边界：写成文档
```

### 4.2 范围蔓延预防
```
范围蔓延预防：

预防措施：
├── 明确验收标准：什么算完成
├── 建立变更流程：范围变更要走流程
├── 做好需求优先级：先做重要的
├── 定期审视范围：检查是否偏离
├── 沟通期望：让干系人知道范围边界
└── 记录范围变更：记录每次变更

范围变更信号：
├── 需求不断增加
├── "顺便加一下"越来越多
├── 原计划时间总是被压缩
├── 团队抱怨需求太多
└── 客户期望越来越膨胀
```

## 5. 启动会准备

### 5.1 启动会准备清单
```
启动会准备：

会议准备：
□ 确定会议时间（提前1周通知）
□ 确定参会人员
□ 确定会议地点/方式
□ 准备会议材料
□ 发送会议邀请
□ 确认关键人出席

材料准备：
□ 项目章程（草稿）
□ 项目目标
□ 项目范围
□ 团队组织
□ 项目计划（初步）
□ 沟通机制

个人准备：
□ 熟悉项目背景
□ 准备开场演讲
□ 准备Q&A问题
□ 准备茶点（如果线下）
```

### 5.2 启动会议程
```
启动会议程（1-2小时）：

环节1：开场（10分钟）
├── 欢迎参会人员
├── 自我介绍
└── 会议目标说明

环节2：项目介绍（20分钟）
├── 项目背景
├── 项目目标
├── 项目价值
└── 成功标准

环节3：范围与计划（30分钟）
├── 项目范围
├── 项目阶段
├── 关键里程碑
└── 资源与预算

环节4：团队介绍（20分钟）
├── 团队成员角色
├── 团队成员介绍
└── 协作方式说明

环节5：沟通与流程（15分钟）
├── 沟通机制
├── 汇报流程
├── 问题升级路径
└── 决策机制

环节6：讨论与问答（20分钟）
├── 开放提问
├── 疑虑解答
├── 共识确认
└── 下一步行动

环节7：结束（5分钟）
├── 总结要点
├── 感谢参会
└── 会后事项
```

### 5.3 启动会演讲要点
```
启动会开场演讲：

要点1：为什么做这个项目
"各位好，今天是我们XX项目的启动会
这个项目源于XX问题/机会
如果不做XX，会有XX影响
如果做成XX，会带来XX价值
"

要点2：项目目标是什么
"我们项目的目标是XX
具体来说，我们要交付XX
项目成功的标准是XX
"

要点3：团队与分工
"这个项目由XX团队负责
我是项目经理XX，负责整体把控
产品负责人是XX，负责需求把控
技术负责人是XX，负责技术实现
"

要点4：需要大家的支持
"项目成功需要大家的配合
希望XX部门在XX方面给予支持
希望XX在XX时间前完成XX
谢谢大家！
"
```

## 6. 启动会执行

### 6.1 启动会注意事项
```
启动会注意事项：

做得好的：
├── 提前准备好所有材料
├── 鼓励团队成员发言
├── 正面回应质疑
├── 明确下一步行动
├── 会后发会议纪要
└── 合影留念（如有必要）

避免的问题：
├── 材料没准备好就上场
├── 一言堂不讲互动
├── 不能回答基本问题
├── 没有明确行动计划
├── 开成批判会
└── 会后不了了之
```

### 6.2 启动会常见问题
```
启动会常见问题及应对：

Q：团队成员不熟悉怎么办？
A：安排自我介绍环节，准备一些破冰问题

Q：有人质疑项目价值怎么办？
A：准备项目背景和价值说明，用数据和事实说话

Q：有人提出大范围需求怎么办？
A：记录但不立即承诺，会后单独讨论

Q：关键人物没来怎么办？
A：提前确认，无法到场要提前沟通，争取视频接入

Q：讨论偏离主题怎么办？
A：温和拉回，记录偏离话题，约定会后讨论
```

### 7.2 常见陷阱
```
项目启动常见陷阱：

陷阱1：启动会开成批斗会
├── 原因：没有准备好就开会
├── 后果：团队抵触，后续难推动
└── 预防：充分准备，正面引导

陷阱2：目标过于理想化
├── 原因：没有充分评估
├── 后果：后面无法达成，团队失去信心
└── 预防：基于数据制定目标

陷阱3：范围定义不清晰
├── 原因：急着开始没定义清楚
├── 后果：范围蔓延，无法交付
└── 预防：明确定义范围，获得认可

陷阱4：干系人没对齐
├── 原因：没有充分沟通
├── 后果：关键时得不到支持
└── 预防：提前一对一沟通
```
