## TL;DR

需求变更 = 变更申请 + 影响评估 + 评审决策 + 实施跟踪 + 总结归档

## 1. 变更认知

### 1.1 什么是需求变更
```
需求变更的定义：
项目启动后，对已确认的需求进行修改、增加、删除的行为

需求变更的来源：
├── 客户/用户：新需求、需求调整
├── 市场：市场变化、政策变化
├── 技术：技术方案调整
├── 团队：实现难度超出预期
├── 业务：业务方向调整
├── 环境：外部依赖变化
└── 发现：测试/用户发现的新需求
```

### 1.2 变更的本质
```
变更的本质：
变更本身不是坏事，合理的变更可以：
├── 让产品更好：基于反馈调整
├── 适应变化：响应市场变化
├── 降低风险：尽早发现问题
└── 满足用户：真正满足用户需求

变更的问题：
├── 失控的变更是项目杀手
├── 频繁变更影响进度
├── 无序变更导致混乱
├── 不评估影响的变更危险
└── 范围蔓延让项目无边界
```

### 1.3 范围蔓延
```
范围蔓延（Scope Creep）：
未经控制的需求范围扩大

蔓延的常见形式：
├── "就加一个小功能"
├── "这个很简单，顺便做了"
├── "用户强烈要求的"
├── "竞品都有了"
└── "这是最后的需求"

蔓延的危害：
├── 项目延期
├── 成本超支
├── 质量下降
├── 团队疲惫
├── 士气低落
└── 项目失败

蔓延的原因：
├── 范围定义不清晰
├── 变更没有走流程
├── 拒绝变更太难
├── 没有评估影响
├── 干系人压力
└── 缺乏变更控制意识
```

## 2. 变更管理流程

### 2.1 变更流程概述
```
变更管理流程（7步法）：

1. 变更申请
   └── 提交人提出变更请求

2. 初步分析
   └── 产品经理初步分析

3. 影响评估
   └── 团队评估影响

4. 变更评审
   └── 评审委员会决策

5. 变更批准/拒绝
   └── 输出评审结论

6. 变更实施
   └── 执行变更

7. 变更验证
   └── 验证变更效果
```

### 2.2 变更申请
```
变更申请要素：

基本信息：
├── 变更编号：唯一标识
├── 申请人：谁提的
├── 申请日期：何时提的
├── 所属项目：哪个项目
└── 优先级：紧急程度

变更内容：
├── 变更类型：新增/修改/删除
├── 涉及模块：哪个功能模块
├── 当前需求：现在是怎么做的
├── 变更需求：希望改成什么
├── 变更原因：为什么要变更
└── 期望日期：希望什么时候完成

业务价值：
├── 业务背景：为什么需要
├── 用户价值：带来什么价值
├── 紧急程度：为什么现在就要
└── 不做的后果：不做的风险

附件：
├── 相关文档
├── 截图/原型
└── 参考材料
```

## 3. 影响评估

### 3.1 影响评估内容
```
影响评估维度：

进度影响：
├── 增加多少工作量
├── 影响哪些里程碑
├── 需要延长多少时间
├── 是否影响上线时间
└── 需要哪些资源

成本影响：
├── 开发成本：多少人天
├── 测试成本：多少测试工作
├── 其他成本：培训、维护
├── 总成本增加
└── 是否在预算内

技术影响：
├── 技术复杂度
├── 架构影响
├── 系统集成影响
├── 技术风险
└── 技术可行性

质量影响：
├── 功能测试影响
├── 性能测试影响
├── 兼容性影响
├── 回归测试范围
└── 质量风险

资源影响：
├── 需要多少人
├── 需要什么技能
├── 是否需要外部资源
├── 对其他项目的影响
└── 资源冲突
```

### 3.3 评估注意事项
```
影响评估注意事项：

评估要客观：
├── 不要为了通过而低估影响
├── 不要为了拒绝而高估影响
├── 基于数据和事实
└── 说明评估依据

评估要全面：
├── 考虑直接影响和间接影响
├── 考虑短期影响和长期影响
├── 考虑对其他部分的影响
└── 不要遗漏重要影响

评估要及时：
├── 收到变更后及时评估
├── 在评审前完成评估
├── 评估有期限（建议1-3天）
└── 紧急变更走快速流程

评估要沟通：
├── 和开发团队确认工作量
├── 和测试团队确认测试量
├── 和相关方确认影响
└── 达成共识后再评审
```

## 4. 变更评审

### 4.1 评审委员会
```
变更评审委员会：

组成成员：
├── 产品负责人：业务决策权
├── 技术负责人：技术可行性
├── 项目经理：进度和资源
├── 测试负责人：质量影响
└── 相关专家：必要时参与

角色职责：
├── 产品负责人：最终决策权
├── 技术负责人：技术把关
├── 项目经理：进度控制
└── 评审秘书：记录和跟踪

决策权设置：
├── 轻微变更：产品经理决策
├── 一般变更：评审委员会决策
├── 重大变更：需要高层参与
└── 紧急变更：授权快速通道
```

### 4.2 评审流程
```
变更评审流程：

1. 评审准备
   ├── 变更申请材料
   ├── 影响评估报告
   ├── 备选方案（如有）
   └── 评审会议通知

2. 评审会议
   ├── 变更申请人陈述
   ├── 产品经理补充
   ├── 评估人说明影响
   ├── 团队讨论
   └── 提问和答疑

3. 评审决策
   ├── 同意变更
   ├── 同意变更（含条件）
   ├── 推迟变更
   ├── 拒绝变更
   └── 要求更多信息

4. 评审输出
   ├── 评审结论
   ├── 变更实施计划
   ├── 责任人分配
   ├── 时间安排
   └── 沟通计划
```

### 4.3 评审决策标准
```
变更评审决策标准：

建议接受的情况：
├── 业务价值高
├── 影响可控
├── 有足够资源
├── 不影响核心目标
└── 变更有充分理由

建议拒绝的情况：
├── 业务价值低
├── 影响范围大
├── 资源不足
├── 影响核心目标
├── 变更理由不充分
└── 频繁变更无序

建议推迟的情况：
├── 价值不确定
├── 影响较大但可优化
├── 当前不是最佳时机
├── 可以放到下一版本
└── 资源暂时不足

决策权重建议：
├── 业务价值：40%
├── 进度影响：20%
├── 成本影响：15%
├── 技术可行性：15%
├── 风险程度：10%
└── 总分>70分建议接受
```

## 6. 变更控制策略

### 6.1 预防范围蔓延
```
预防范围蔓延策略：

明确定义范围：
├── 需求文档清晰
├── 验收标准明确
├── 范围边界清楚
├── 排除项列明
└── 获得各方确认

建立变更机制：
├── 变更必须走流程
├── 变更必须有评估
├── 变更必须有决策
├── 变更必须有记录
└── 变更必须有跟踪

控制变更频率：
├── Sprint内不变更
├── 控制单次变更数量
├── 设置变更窗口
├── 分析变更趋势
└── 及时发现异常

学会说"不"：
├── 用数据说话
├── 说明变更成本
├── 提供替代方案
├── 升级决策
└── 坚持原则
```

### 6.2 变更分级
```
变更分级管理：

轻微变更（紧急变更）：
├── 定义：工作量<1人天，不影响其他需求
├── 审批人：产品经理
├── 流程：简化流程，快速决策
├── 时限：当天决策
└── 示例：文案修改、小bug修复

一般变更：
├── 定义：工作量1-5人天，影响有限
├── 审批人：评审委员会
├── 流程：标准流程
├── 时限：1-3天决策
└── 示例：功能优化、小功能增加

重大变更：
├── 定义：工作量>5人天，影响较大
├── 审批人：高层+评审委员会
├── 流程：完整评审流程
├── 时限：3-5天决策
└── 示例：大功能增加、架构调整

紧急变更：
├── 定义：线上问题、安全漏洞
├── 审批人：技术负责人
├── 流程：紧急通道
├── 时限：立即处理
└── 要求：事后补齐流程
```

## 7. 变更管理常见问题

### 7.1 常见问题处理
```
问题1：业务方频繁变更

原因：
├── 需求一开始就没想清楚
├── 业务方向不确定
├── 沟通不充分
└── 没有建立变更机制

应对：
├── 需求阶段充分沟通
├── 原型确认后再开发
├── 建立变更门槛
├── 透明变更成本
└── 升级沟通层级

问题2：开发人员擅自变更

原因：
├── 变更没有走流程
├── 沟通不及时
├── 缺乏规范
└── 变更太容易

应对：
├── 建立变更流程
├── 强调变更规范
├── 代码评审检查
├── 定期范围检查
└── 明确责任

问题3：拒绝变更导致业务损失

原因：
├── 过于保守
├── 拒绝合理变更
├── 流程过于复杂
└── 缺乏灵活性

应对：
├── 建立分级变更机制
├── 轻微变更快速决策
```
