## TL;DR

敏捷迭代 = Sprint规划 + 每日站会 + 迭代评审 + 回顾改进

## 1. 敏捷基础认知

### 1.1 敏捷核心价值观
```
敏捷宣言：
├── 个体和互动 > 流程和工具
├── 可工作的软件 > 详尽的文档
├── 客户合作 > 合同谈判
└── 响应变化 > 遵循计划

敏捷原则：
├── 最高优先级：早期和持续交付有价值的软件
├── 欢迎变化：即使在开发后期也欢迎需求变化
├── 频繁交付：频繁交付可工作的软件
├── 每日协作：业务人员和开发者每日协作
├── 激励团队：建立信任，赋予团队自主权
├── 面对面沟通：最有效的沟通方式
├── 可工作软件：可工作的软件是进度的主要衡量
├── 可持续开发：各方都能保持长期稳定的节奏
├── 追求卓越：持续关注技术卓越和设计质量
├── 简洁：减少不必要的工作
├── 自组织团队：最好的架构、需求和设计来自自组织团队
└── 定期反思：团队定期反思如何提升效率
```

### 1.2 常见敏捷框架
```
常见敏捷框架：

Scrum：
├── 角色：Product Owner、Scrum Master、开发团队
├── 事件：Sprint、Sprint Planning、Daily Scrum、Sprint Review、Sprint Retrospective
├── 工件：Product Backlog、Sprint Backlog、Increment
└── 特点：固定长度的迭代（通常1-4周）

Kanban：
├── 核心：可视化、限制在制品、流动管理
├── 看板：看板墙、卡片、列
├── 策略：持续交付，不固定迭代
└── 特点：灵活性高，适合维护型团队

XP（极限编程）：
├── 实践：结对编程、持续集成、测试驱动开发
├── 价值观：沟通、简单、反馈、勇气
└── 特点：技术实践导向
```

### 1.3 什么时候适合敏捷
```
适合敏捷的场景：
├── 需求不确定或经常变化
├── 需要快速响应市场
├── 跨职能团队协作
├── 产品需要持续迭代
├── 技术风险较高的项目
└── 需要早期验证假设

不太适合敏捷的场景：
├── 需求非常稳定明确
├── 有明确的法规/合规要求
├── 一次性交付，无后续迭代
├── 团队成员经验不足
└── 组织文化不支持
```

## 2. Sprint规划

### 2.1 Sprint规划准备
```
Sprint规划准备：

Product Owner准备：
├── Product Backlog已准备就绪
├── backlog条目已按优先级排序
├── 高优先级条目有清晰的描述
├── 验收标准已定义
└── 业务价值已说明

团队准备：
├── 了解历史迭代的速度
├── 了解当前团队能力
├── 了解可能的障碍
└── 准备好讨论

环境准备：
├── 确定Sprint时长（1-4周）
├── 确定Sprint目标
├── 准备估算工具
└── 预留足够时间（2-4小时）
```

### 2.2 Sprint规划流程
```
Sprint规划流程（2-4小时）：

Part 1：Sprint目标（30分钟）
├── Product Owner介绍Sprint目标
├── 讨论目标的背景和价值
├── 团队和PO共同确认目标
└── 记录Sprint目标

Part 2：选择Backlog（1-2小时）
├── 从顶部开始，逐条讨论
├── 团队评估能否完成
├── 确认验收标准
├── 添加到Sprint Backlog
└── 直到团队容量足够

Part 3：任务分解（1-2小时）
├── 将Story分解为Task
├── 团队成员主动领取任务
├── 估算Task的工作量
├── 识别依赖和风险
└── 确定Sprint计划
```

### 2.3 Sprint容量估算
```
容量估算方法：

故事点数估算：
├── 相对估算：与其他故事比较
├── 斐波那契数列：1,2,3,5,8,13,21
├── 团队估算：开发团队一起估算
├── 参考速率：基于历史迭代速率
└── 不要承诺超出容量的工作

速率计算：
├── 历史Sprint速率：平均每个Sprint完成X个故事点
├── 当前Sprint容量：根据可用人力调整
├── 安全边际：预留10-20%buffer
└── 最终承诺：基于实际能力承诺

容量规划表：
| 团队成员 | 可用天数 | 备注 |
|----------|----------|------|
| 成员A   | 8天     | 请假2天 |
| 成员B   | 10天    | 全勤 |
| ... | ... | ... |
| 合计   | X天     | 团队总容量 |
```

## 3. 每日站会

### 3.1 站会目的
```
每日站会的目的：
├── 同步进展：让团队知道每个人在做什么
├── 识别障碍：发现阻碍进度的障碍
├── 促进协作：发现需要互相配合的地方
├── 保持专注：提醒团队Sprint目标
└── 快速决策：需要帮助时立即决策
```

### 3.2 站会怎么开
```
站会标准流程（15分钟）：

站会规则：
├── 每天同一时间
├── 站立开会（保持简短）
├── 每个人都要说
├── 专注在Sprint目标
├── 不解决问题（会后单独讨论）

每人发言模板：
"昨天完成了XX
今天计划做XX
遇到的障碍是XX（如果有）"

主持技巧：
├── 按固定顺序发言
├── 控制时间，不展开讨论
├── 记录障碍，会后跟进
├── 鼓励团队互相帮助
└── 保持积极氛围
```

### 3.3 站会问题处理
```
站会常见问题：

问题1：站会变成汇报会
├── 原因：领导追问细节
├── 解决：提醒只说进展，不展开
└── 技巧：用"会后找我"转移

问题2：讨论变成会议
├── 原因：有人开始详细讨论
├── 解决："这个问题我们站后讨论"
└── 技巧：记下来，站后组织讨论

问题3：有人迟到/缺席
├── 原因：各种原因
├── 解决：提前提醒，明确规则
└── 技巧：了解原因，针对处理

问题4：障碍一直存在
├── 原因：障碍没有被跟进解决
├── 解决：SM要主动跟进障碍
└── 技巧：列出障碍清单，逐个解决
```

## 4. Sprint评审

### 4.1 评审准备
```
Sprint评审准备：

团队准备：
├── 完成所有承诺的故事
├── 确保软件可演示
├── 准备演示环境
├── 准备演示数据
└── 内部预演（必要时）

Product Owner准备：
├── 准备评审议程
├── 邀请关键干系人
├── 准备反馈收集方式
└── 准备讨论话题

材料准备：
├── Sprint回顾：做了什么
├── 演示内容：展示什么
├── 数据支持：演示数据
└── 问题清单：可能的问题
```

### 4.2 评审怎么开
```
Sprint评审流程（1-2小时）：

开场（10分钟）：
├── 欢迎参会者
├── 说明评审目标
├── 介绍Sprint目标
└── 简要回顾Sprint

演示（30-60分钟）：
├── 演示完成的功能
├── 演示实际使用场景
├── 展示用户故事完成情况
├── 演示bug修复
└── 回答问题

讨论（30分钟）：
├── 收集反馈
├── 讨论遇到的问题
├── 澄清疑问
└── 讨论下一步

结束（10分钟）：
├── 总结反馈
├── 记录后续行动
└── 感谢参会
```

### 4.3 评审注意事项
```
评审注意事项：

做得好的：
├── 聚焦在可工作的软件
├── 演示真实使用场景
├── 鼓励干系人提问
├── 记录所有反馈
├── 讨论下一步行动
└── 控制时间

避免的问题：
├── 演示半成品
├── 只讲技术不讲价值
├── 忽视干系人反馈
├── 开成验收会
├── 演示环境不稳定
└── 超时太多
```

## 5. Sprint回顾

### 5.1 回顾准备
```
回顾准备：

数据收集：
├── Sprint完成情况：承诺vs实际
├── 燃尽图趋势：是否按计划进行
├── 质量问题：bug数量、线上问题
├── 团队状态：士气、协作情况
└── 障碍记录：本Sprint遇到的障碍

材料准备：
├── 数据可视化：图表、趋势
├── 便利贴：用于书写想法
├── 回顾工具：可以选择回顾方法
└── 零食饮料：营造轻松氛围

时间安排：
├── Sprint结束后24-48小时内
├── 预留2-3小时
├── 选择安静舒适的场地
└── 全团队参与
```

### 5.2 回顾方法
```
常见回顾方法：

4L回顾法：
├── Liked：喜欢的
├── Learned：学到的
├── Lacked：缺乏的
└── Longed for：期望的

Start-Stop-Continue：
├── Start：开始做什么
├── Stop：停止做什么
└── Continue：继续做什么

 Sailboat回顾（帆船回顾）：
├── Wind（风）：推动我们前进的
├── Anchor（锚）：拖住我们的
├── Rocks（礁石）：可能让我们搁浅的
└── Island（岛）：我们到达的里程碑

Mad Sad Glad回顾：
├── Mad：让我们生气的
├── Sad：让我们难过的
└── Glad：让我们高兴的
```

### 5.3 回顾执行
```
回顾执行流程（2-3小时）：

开场（10分钟）：
├── 感谢团队
├── 说明回顾目的
├── 回顾上次改进项
└── 选择回顾方法

数据回顾（20分钟）：
├── 展示Sprint数据
├── 讨论趋势
├── 识别好的/不好的
└── 记录观察

头脑风暴（30分钟）：
├── 按回顾方法分类
├── 每人写出想法
├── 匿名收集（可选）
└── 不讨论，只收集

讨论优先排序（30分钟）：
├── 解释每个想法
├── 合并相似想法
├── 团队投票
└── 确定top 2-3项

制定改进计划（30分钟）：
├── 针对top问题
├── 讨论解决方案
├── 指定负责人
├── 确定完成时间
└── 下次回顾检查

结束（10分钟）：
├── 感谢参与
├── 总结改进项
└── 会后发送纪要
```

## 6. 迭代常见问题

### 6.1 需求做不完
```
需求做不完处理：

原因分析：
├── 估算不准确
├── 需求理解偏差
├── 临时加入需求
├── 人员变动
└── 技术问题

应对措施：
├── 调整Sprint范围：和PO协商，移出低优先级story
├── 加班（谨慎）：如果是因为团队问题
├── 延长Sprint（特殊情况）：但要反思
└── 改进估算：下次迭代更准确

预防措施：
├── 准确估算：基于历史数据
├── 保留buffer：预留10-20%容量
├── 控制变更：Sprint内不变更
├── 及时识别：尽早发现完不成
└── 透明沟通：及时和PO沟通
```

### 6.2 质量不达标
```
质量不达标处理：

表现：
├── bug太多
├── 代码质量差
├── 线上事故频发
└── 返工率高

原因：
├── 赶进度牺牲质量
├── 没有足够测试
├── 技术债累积
└── 缺少代码review

改进措施：
├── 增加测试时间
├── 引入代码review
├── 技术债专项迭代
├── 增加自动化测试
└── 质量指标透明化

预防措施：
├── 定义质量标准
├── 持续集成
├── 自动化测试
├── 技术实践培训
└── 团队质量意识
```

### 6.3 团队协作问题
```
团队协作问题处理：

问题1：成员参与度低
├── 表现：站会不说话，不主动认领任务
├── 原因：缺乏归属感、技能不足、人际关系
├── 解决：私下沟通，了解原因，针对性帮助

问题2：成员之间协作差
├── 表现：各自为战，不互相帮助
├── 原因：缺乏共同目标、责任不清
├── 解决：强调团队目标，明确协作方式

问题3：团队士气低
├── 表现：抱怨多、负能量、离职倾向
├── 原因：工作压力大、认可不足、职业发展
├── 解决：一对一沟通，了解诉求，改进管理

问题4：跨团队协作困难
├── 表现：依赖方响应慢、推诿
├── 原因：优先级不同、缺乏沟通
├── 解决：高层沟通、建立合作关系、明确责任
```

## 7. 敏捷度量

### 7.1 常用度量指标
```
敏捷度量指标：

速度（Velocity）：
├── 定义：每个Sprint完成的故事点数
├── 用途：估算未来Sprint容量
├── 计算：Sprint完成的故事点总和/迭代数
└── 注意：不要用速度来施压团队

燃尽图（Burndown Chart）：
├── 定义：剩余工作量随时间的变化
├── 用途：跟踪Sprint进度
├── 理想线：直线下降
└── 解读：偏离理想线说明有问题

累积流图（Cumulative Flow Diagram）：
├── 定义：各类工作随时间的累积
├── 用途：识别瓶颈
└── 解读：某列增长说明瓶颈在该环节

缺陷趋势：
├── 定义：缺陷数量随时间的变化
├── 用途：监控质量
└── 解读：上升说明质量有问题
```

### 7.2 度量注意事项
```
度量注意事项：

避免的误区：
├── 速度排名：用速度施压团队
├── 微观管理：用度量控制团队
├── 短期视角：只看单迭代数据
├── 数据造假：为了好看的数字
└── 过度度量：度量太多反而无效

正确的态度：
├── 透明公开：数据对团队可见
├── 持续改进：数据用于改进
├── 团队拥有：团队参与定义度量
├── 关注趋势：看长期趋势
└── 理解数据：理解数据的含义
```
