## TL;DR

风险管理 = 风险识别 + 风险评估 + 风险应对 + 风险监控

## 1. 风险管理认知

### 1.1 什么是风险
```
风险的定义：
可能发生的不利事件，一旦发生会对项目目标产生负面影响

风险的特点：
├── 不确定性：可能发生也可能不发生
├── 影响两面：可以是威胁也可以是机会
├── 可管理：通过措施可以降低发生概率或影响
├── 动态变化：风险会随着项目进展变化
└── 普遍存在：每个项目都有风险
```

### 1.2 风险分类
```
风险分类：

按来源分类：
├── 技术风险：技术难度、技术选型、技术变更
├── 人员风险：人员流失、人员能力、人员冲突
├── 需求风险：需求变更、需求不清、需求遗漏
├── 资源风险：资源不足、资源延迟、资源变更
├── 外部风险：市场变化、政策变化、供应商问题
├── 管理风险：计划不当、沟通不畅、决策失误
└── 业务风险：业务方向变化、优先级调整

按影响分类：
├── 进度风险：影响项目时间
├── 成本风险：影响项目预算
├── 质量风险：影响交付质量
├── 范围风险：影响项目范围
└── 声誉风险：影响公司/产品声誉
```

### 1.3 风险管理的重要性
```
风险管理的重要性：
├── 预防问题：提前识别，提前准备
├── 减少损失：降低风险发生时的损失
├── 提高成功率：风险管理好的项目成功率更高
├── 增强信心：知道风险在哪里，心里有底
├── 资源优化：合理分配资源应对风险
└── 赢得信任：让干系人看到你的专业性
```

## 2. 风险识别

### 2.1 风险识别方法
```
风险识别方法：

头脑风暴：
├── 组织团队一起识别风险
├── 每个人提出自己的想法
├── 不批评，鼓励发散
├── 记录所有风险
└── 会后整理归类

检查表法：
├── 使用历史项目的风险清单
├── 参考行业风险数据库
├── 根据项目特点增减
└── 逐项检查

SWOT分析：
├── Strengths（优势）- 优势如何变成风险
├── Weaknesses（劣势）- 劣势带来什么风险
├── Opportunities（机会）- 机会伴随什么风险
└── Threats（威胁）- 直接威胁是什么

假设分析：
├── 识别项目假设
├── 分析假设不成立的风险
├── 检验假设的合理性
└── 制定应急计划

专家访谈：
├── 访谈有经验的人
├── 听取专家意见
├── 参考行业经验
└── 综合各方观点

分解结构：
├── WBS分解
├── 逐层分析风险
├── 识别各层级的风险
└── 汇总整理
```

## 3. 风险评估

### 3.1 定性评估
```
定性风险评估：

发生概率评估：
├── 极高（>80%）：很可能发生
├── 高（60-80%）：很可能发生
├── 中（40-60%）：可能发生
├── 低（20-40%）：不太可能发生
└── 极低（<20%）：几乎不可能发生

影响程度评估：
├── 极高：项目完全失败
├── 高：严重影响项目目标
├── 中：对项目有显著影响
├── 低：对项目影响有限
└── 极低：影响可忽略

风险等级矩阵：
         极低影响  低影响  中影响  高影响  极高影响
极高概率  中风险   高风险  高风险  极高风险  极高风险
高概率   低风险   中风险  高风险  极高风险  极高风险
中概率   低风险   中风险  中风险  高风险   极高风险
低概率   低风险   低风险  中风险  高风险   高风险
极低概率 低风险   低风险  低风险  中风险   中风险
```

### 3.2 定量评估
```
定量风险评估方法：

蒙特卡洛模拟：
├── 定义变量和概率分布
├── 模拟大量随机场景
├── 统计分析结果分布
├── 计算风险值（VaR）
└── 适用于：成本/进度风险

敏感性分析：
├── 识别关键变量
├── 逐个变量变化分析
├── 确定对结果的影响
├── 找出最敏感的变量
└── 适用于：关键决策

决策树分析：
├── 识别决策点和风险点
├── 计算各路径的期望值
├── 比较各方案优劣
├── 选择最优方案
└── 适用于：方案选择

预期货币价值（EMV）：
├── 列出所有可能结果
├── 计算每个结果的概率
├── 计算预期货币价值
├── 考虑风险偏好
└── 适用于：投资决策
```

### 3.3 风险优先级排序
```
风险优先级排序：

优先级评估标准：
├── 风险值 = 发生概率 × 影响程度
├── 风险等级：高/中/低
├── 紧迫程度：时间敏感性
├── 可控程度：管理难度
├── 连锁效应：是否引发其他风险
└── 干系人关注：重要干系人关注度

优先级排序表：
| 风险 | 概率 | 影响 | 风险值 | 等级 | 优先级 |
|------|------|------|--------|------|--------|
| R1  | 高   | 高   | 高     | 高   | P1    |
| R2  | 中   | 高   | 中     | 高   | P2    |
| R3  | 高   | 中   | 中     | 中   | P3    |
| R4  | 低   | 中   | 低     | 中   | P4    |
| ... | ... | ... | ...   | ... | ...   |

处理优先级：
├── P1：立即处理，需要高层关注
├── P2：本周处理，项目经理跟进
├── P3：本月处理，责任到人
└── P4：监控即可，定期检查
```

## 4. 风险应对

### 4.1 应对策略
```
四大风险应对策略：

1. 规避（Avoid）
├── 定义：消除风险发生的原因
├── 适用：高概率 + 高影响
├── 方法：改变计划、取消项目、改变范围
└── 示例：更换技术方案、取消高风险功能

2. 转移（Transfer）
├── 定义：将风险转移给第三方
├── 适用：高概率或高影响，但可以转移
├── 方法：保险、外包、合同条款、期权
└── 示例：购买保险、外包高风险模块、签订赔偿条款

3. 减轻（Mitigate）
├── 定义：降低风险概率或影响
├── 适用：中/高概率 + 中/高影响
├── 方法：预防措施、应急计划、培训
└── 示例：增加测试、备份方案、人员培训

4. 接受（Accept）
├── 定义：接受风险，准备应急
├── 适用：低概率 + 低影响，或无法处理
├── 方法：建立应急储备、制定应急计划
└── 示例：预留时间和预算、建立监控机制
```

### 4.2 应对计划制定
```
风险应对计划：

计划内容：
├── 风险描述
├── 应对策略
├── 具体措施
├── 责任人
├── 开始/结束时间
├── 资源需求
├── 预算
├── 触发条件
├── 应急计划
└── 残余风险

计划示例：
风险：核心开发人员离职
策略：减轻
具体措施：
  1. 交叉培训，确保至少2人熟悉核心模块
  2. 建立知识库，记录关键技术文档
  3. 招聘备份人员
责任人：技术负责人
时间：持续进行
资源：培训时间、招聘费用
触发条件：核心人员提出离职
应急计划：立即启动招聘，同时分配任务给备份人员
残余风险：人员离职初期可能有短暂影响
```

### 4.3 应对措施库
```
不同风险类型的应对措施：

技术风险应对：
├── 技术难度高 → POC验证、增加培训、引入专家
├── 技术变更 → 技术调研、架构评审、兼容性考虑
├── 集成问题 → 早期集成测试、接口协议明确
└── 性能问题 → 性能测试、容量规划、优化方案

需求风险应对：
├── 需求变更 → 变更流程、影响分析、优先级重排
├── 需求不清 → 需求澄清会、原型确认、用户测试
└── 需求遗漏 → 需求评审、测试覆盖、用户反馈

资源风险应对：
├── 人员不足 → 招聘计划、借调、加班（谨慎）
├── 预算削减 → 范围重排、优先级调整、寻求额外预算
└── 设备延迟 → 提前采购、替代方案、风险准备

外部风险应对：
├── 市场变化 → 敏捷响应、AB测试、快速迭代
├── 政策变化 → 合规咨询、预留时间、监控政策
└── 供应商问题 → 备份供应商、合同保障、替代方案
```

## 5. 风险监控

### 5.1 监控机制
```
风险监控机制：

定期审查：
├── 每周：检查高优先级风险
├── 每两周：检查所有风险状态
├── 每月：风险再评估，更新登记册
├── 里程碑：关键节点风险评审
└── 触发审查：触发条件满足时

持续监控：
├── 风险指标监控
├── 早期预警信号
├── 风险趋势分析
├── 对比计划vs实际
└── 沟通汇报

监控责任人：
├── 项目经理：整体风险监控
├── 风险责任人：各自风险跟踪
├── 团队成员：发现新风险及时上报
└── 干系人：定期风险汇报
```

### 5.2 风险预警
```
风险预警信号：

进度风险信号：
├── 任务持续延期
├── 燃尽图偏离
├── 里程碑临近但未完成
├── 返工率高
└── 团队产出下降

质量风险信号：
├── bug数量增加
├── 测试覆盖率下降
├── 线上问题增多
├── 代码review问题多
└── 技术债务累积

资源风险信号：
├── 人员加班过多
├── 关键人员表现异常
├── 设备/工具问题
├── 预算消耗过快
└── 外部依赖延迟

沟通风险信号：
├── 会议参与度下降
├── 反馈变少
├── 冲突增加
├── 干系人关注增加
└── 负面反馈增多
```

### 5.3 风险应对启动
```
风险应急启动：

启动条件：
├── 风险即将发生
├── 风险已经发生
├── 风险征兆出现
├── 触发条件满足
└── 超出容忍范围

启动流程：
1. 确认风险状态
2. 评估当前影响
3. 启动应急计划
4. 通知相关干系人
5. 分配应急资源
6. 执行应对措施
7. 持续跟踪
8. 恢复正常状态
9. 总结经验教训

注意事项：
├── 不要等风险发生才行动
├── 及时升级，不要隐瞒
├── 快速决策，不要犹豫
├── 透明沟通，不要隐瞒
└── 记录过程，便于复盘
```

## 6. 风险管理实践

### 6.1 项目各阶段风险管理
```
项目各阶段风险管理重点：

启动阶段：
├── 识别战略风险
├── 评估项目可行性
├── 识别关键假设风险
├── 建立风险管理机制
└── 制定风险应对计划

规划阶段：
├── 详细识别各领域风险
├── 评估技术风险
├── 识别资源风险
├── 制定详细应对计划
└── 建立风险监控机制

执行阶段：
├── 监控风险状态
├── 执行应对措施
├── 识别新风险
├── 更新风险登记册
└── 定期风险评审

收尾阶段：
├── 确认风险关闭
├── 总结风险经验
├── 归档风险文档
├── 风险知识沉淀
```
